# RCONS: детальный план рефакторинга (4 спринта)

Цель: стабилизировать архитектуру сервиса аудита/консалтинга под Next.js App Router, убрать типовые ошибки, подготовить SEO/поиск/AI-чат/лид-форму под RU-рынок и требования безопасности 2026.

---

## 0) Базовые решения (фиксируем до начала работ)

1. `globals.css` оставляем как есть по имени (это корректный стандарт Next.js).
2. Источник переводов: `messages/ru.json` (source of truth), остальные локали — синхронизация + черновой автоперевод + ручная QA.
3. Иерархия услуг фиксируется строго:
   - `/[locale]/services` → категории;
   - `/[locale]/services/[category]` → список типов/услуг категории;
   - `/[locale]/services/[category]/[type]` → детальная страница услуги.
4. Никаких `as any` в сервисных страницах, i18n и доменных конфигах.
5. Для AI-чата и поиска используем только внутренний контент (services/pricing/messages/pages/resources/documents).

## 1) Архитектурный каркас (что меняем в коде)

### 1.1 Маршруты услуг

- `src/app/[locale]/services/page.tsx`
  - оставить только каталог категорий;
  - убрать ожидание `category/type` в `params`;
  - ссылки только на `/${locale}/services/${category}`.

- `src/app/[locale]/services/[category]/page.tsx`
  - оставить только список услуг/типов категории;
  - ссылки только на `/${locale}/services/${category}/${type}`;
  - при невалидном category → `notFound()`.

- `src/app/[locale]/services/[category]/[type]/page.tsx`
  - оставить detail-логику: hero, что делаем, price card, процесс, faq, CTA;
  - при невалидных category/type → `notFound()`.

### 1.2 Типизация и строгие контракты

Вводим единые типы:
- `Locale`
- `ServiceId`
- `ServiceCategorySlug`
- `ServiceTypeSlug`
- `PricingMode = 'fixed' | 'from' | 'on_request'`

Обязательные хелперы:
- `getCategoryBySlug(slug): ServiceCategory | null`
- `getTypeBySlug(categorySlug, typeSlug): ServiceType | null`
- `getServiceById(id): Service | null`

### 1.3 Нормализация конфигов

- `src/config/services.ts`
  - добавить `serviceId`;
  - добавить `searchKeywordsRu`, `intentTags`, `industryTags`;
  - добавить `seoTitleKey`, `seoDescriptionKey`;
  - выровнять slugs, чтобы они 1:1 совпадали с роутами.

- `src/config/pricing.ts`
  - хранить прайс по `serviceId`;
  - структура: `pricingMode`, `amount`, `currency`, `unit`, `priceOnRequest`;
  - запретить логику цен вне этого конфига.

- `src/config/pages.ts`
  - унификация page metadata: `titleKey`, `descriptionKey`, `ogImage`, `canonicalPath`.

## 3) SEO, поиск, AI-чат, форма (RU-friendly)

### 3.1 Поиск

MVP:
- локальный индекс в TS/JSON;
- ранжирование: `exact > title > keywords > intentTags`;
- выдача с типом результата: категория/услуга/ресурс.

Scale:
- при росте каталога — self-hosted OpenSearch/Meilisearch в локальном контуре.

### 3.2 SEO

- метаданные генерируем из конфигов;
- отдельные `title/description` для category/type страниц;
- хлебные крошки + перелинковка related services;
- контент-блоки без заглушек.

### 3.3 AI-чат

- retrieval только из внутренних источников проекта;
- глобальное понимание контента сайта: услуги, документы, статьи, позиционирование компании;
- ответы по ценам только из `pricing`;
- запрет категоричных правовых/налоговых выводов без дисклеймера.

### 3.4 Форма обратной связи

- отправка на email компании через server route;
- серверная валидация (zod/аналог);
- honeypot + rate limit + anti-bot;
- отдельное исключение для легитимных поисковых ботов (Yandex/Google crawl).

### 3.5 Безопасность и комплаенс 2026

- только env для секретов;
- no secret leakage в клиентский бандл;
- минимизация и контроль логирования ПДн;
- явное согласие 152-ФЗ;
- хранить только минимально необходимое по заявке.

## 6) 4 спринта (кратко и прагматично)

### Спринт 1 — Стабилизация ядра (критический)

Делаем:
- выпрямляем 3 сервисных роута по ответственности;
- убираем `as any` в сервисном контуре и i18n-ключах;
- фиксируем доменные типы и safe-хелперы;
- приводим navbar/sidebar к корректной иерархии.

Результат спринта:
- архитектура услуг логически правильная;
- меньше скрытых TS-ошибок;
- навигация соответствует UX “категория → услуга → деталка”.

### Спринт 2 — Данные, SEO, поиск

Делаем:
- нормализуем `services.ts` + `pricing.ts` через `serviceId`;
- добавляем SEO-поля и генерацию metadata;
- внедряем MVP-поиск с ранжированием и подсказками.

Результат спринта:
- единый source of truth;
- предсказуемый SEO-контур;
- рабочий внутренний поиск по интентам.

### Спринт 3 — AI-чат, форма, безопасность

Делаем:
- контекстный AI-чат с retrieval только из внутренних данных;
- запреты/дисклеймеры для правовых и ценовых ответов;
- contact form: валидация, антиспам, 152-ФЗ, email-пайплайн;
- secret policy и hardening логирования.

Результат спринта:
- безопасный и полезный чат;
- рабочий лид-канал без мусорных заявок;
- соответствие RU-комплаенсу.

### Спринт 4 — Контент и полировка UX

Делаем:
- заменяем заглушки на полноценный текст для услуг;
- внедряем единые токены/состояния в `globals.css` + shadcn-компоненты;
- доводим FAQ/процесс/CTA на всех ключевых страницах.

Результат спринта:
- готовый к запуску B2B-сайт без “пустых” страниц;
- консистентный визуальный язык;
- понятный путь пользователя к заявке.

## 7) Definition of Done

1. Иерархия `/services -> /services/[category] -> /services/[category]/[type]` полностью соблюдена.
2. Нет `as any` в сервисном, SEO и i18n-контуре.
3. Поиск находит по названиям, интентам и ключевым словам.
4. AI-чат отвечает по контексту сайта и соблюдает дисклеймеры.
5. Контактная форма безопасна, антиспам включён, есть согласие 152-ФЗ.
6. На ключевых услугах нет заглушек, есть процесс + FAQ + CTA.
